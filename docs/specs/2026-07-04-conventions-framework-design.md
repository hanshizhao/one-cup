# 业务与交互模式标准框架（Conventions Framework）— 设计文档

> 日期：2026-07-04
> 状态：已与用户对齐四段设计，待复核
> 关联：本框架是对 `AGENTS.md` 现有规范体系的补充，不替代 ADR / 技能 / 模板。

---

## 1. 背景与问题

### 1.1 痛点：跨会话一致性漂移

ZCode 在不同会话中处理相似问题时，会采取不同方案。用户反馈的两个典型例子：

1. **删除确认方式不一**：同类"删除"操作，有时用 Popconfirm（气泡），有时用 Modal.confirm（弹窗）。
2. **编号对象创建流程不一**：创建带编号的业务对象（颜色/客户/商品等）时，没有统一走"获取预览编号 → 只读展示 → 无规则则禁用表单 + 提示"的固定流程。

### 1.2 根因诊断

这两个例子暴露的是**同一个缺失**：项目缺少一层"遇到某类业务/交互场景，必须走哪条固定路径"的决策标准。现有体系各管一段，恰好没覆盖这一层：

| 已有手段 | 覆盖层 | 是否覆盖本痛点 |
| --- | --- | --- |
| `AGENTS.md` | 导航架构（Menu/Tabs）、列表查询页布局 | ❌ 不覆盖删除确认、编号流程 |
| `docs/adr/` | 技术栈选型（.NET / PG / Arco） | ❌ 不覆盖业务交互模式 |
| `docs/specs/templates/` | 列表查询页代码模板 | ❌ 仅列表页布局 |
| `.zcode/skills/` | 通用领域执行规范 | ❌ 无项目特定的业务模式约定 |

**结论**：缺的是中间一层——**业务与交互模式标准（Conventions）**。且这类标准目前"只存在于某次对话里"，没有沉淀成可复用的持久化文档。

### 1.3 现状勘查（真实代码印证）

探查现有代码确认漂移确实存在：

- **编号预览**：后端 `GET /api/numbering/preview`（`backend/src/OneCup.Api/Controllers/NumberingController.cs`），前端 `previewCode(targetType, categoryCode?)`（`frontend/src/api/numbering.ts`）。
  - `frontend/src/pages/master-data/color/index.tsx` **已是规范实现**：预览 → 编号 `readOnly` → 无规则时 `noRule=true` → Form `disabled` + Alert 警告 + 确定按钮禁用。
  - `frontend/src/pages/business/customer/form.tsx` 是同一模式的第二实例，实现一致。
- **删除确认**：
  - 角色（`system/role/index.tsx`）删除用 **Popconfirm**。
  - 客户（`business/customer/index.tsx`）删除用 **Modal.confirm**——**全项目唯一一处偏离**。
  - 其余状态切换（color / unit / user / numbering.dict）全部统一用 Popconfirm。

## 2. 设计目标

1. **解决跨会话漂移**：让"遇到 X 场景走 Y 路径"成为不可绕过的约束，而非"我意识到才执行"。
2. **可低门槛扩展**：用户"边用边补"，新增一条标准的门槛低到"加一行表 + 写个决策树"。
3. **不依赖用户看代码**：用户能从 Agent 的回复（标准 ID 引用）核对走的是哪条路径。
4. **复用已验证范式**：不另起炉灶，沿用项目里"列表查询页标准"已跑通的三层结构。

---

## 3. 方案选择

提出过三个方案，选择 A：

| 方案 | 思路 | 取舍 |
| --- | --- | --- |
| **A（采纳）** | AGENTS.md 主决策表 + `docs/conventions/` 详情外链 | ✅ 复用已验证范式；主表每条只占一行，token 便宜；详情可无限扩展 |
| B | 项目级技能 `conventions`，按需加载 | ❌ **致命弱点**：技能靠"我意识到场景该加载"才触发，而漂移恰恰发生在"我没意识到"时——把标准放进需主动判断才加载的地方，等于放大弱点 |
| C | 全部内联进 AGENTS.md | ❌ 不可扩展；每条几百字，加 10 条即臃肿，与"边用边补"冲突 |

**核心论据**：痛点根源是"我没意识到该走标准"。"按需加载"（技能）依赖我意识到才触发；只有 AGENTS.md 自动注入（必然读到）才对症。

---

## 4. 整体架构

复用项目里"列表查询页标准"已验证的三层范式，承载业务/交互模式标准：

```
AGENTS.md（会话启动自动注入 —— 保证必然读到）
  └─「业务与交互模式标准」章节 = 一张主决策表（每条标准一行）
       └─ 每行指向 docs/conventions/c0X-<name>.md（详情，可无限扩展）
```

### 主决策表样例

```markdown
## 业务与交互模式标准（Business & Interaction Conventions）

> 本表是"遇到 X 场景必须走 Y 路径"的索引。实现任何涉及下列场景的功能前，
> 必须先读对应详情文档；改动已有相关功能时，顺手对齐。

| ID | 触发场景 | 必须走的标准 | 详情 |
|----|---------|-------------|------|
| C01 | 任何"删除"操作（行内/批量/详情页） | 按"影响范围 + 可逆性"选 Popconfirm 或 Modal | docs/conventions/c01-delete-confirm.md |
| C02 | 创建"带编号"的业务对象（颜色/客户/商品等） | 固定走"取预览→只读展示→null则禁用+提示"流程 | docs/conventions/c02-numbered-object-create.md |
```

设计要点：
- **ID 用 C 序号**（C = Convention），与 ADR 的序号体系平行，引用方便（"这违反了 C01"）。
- **每条只占一行**：触发场景 + 一句话标准 + 链接。表本身不增长篇幅，增长的是行数。
- **表上方那句"实现前必读、改动时对齐"是强约束语句**——这是机制能否落实的关键。

## 5. 详情文档模板

每条详情文档（`docs/conventions/c0X-<name>.md`）固定四节：

```markdown
# C0X — <标准名>

> 一句话标准：<用一句话概括这条标准>

## 1. 什么时候适用 / 什么时候不适用
<触发条件 + 排除条件，明确边界，避免误判该不该用>

## 2. 标准（决策树 / 固定流程）
<决策树形式：遇到 X → 走 Y。这是消除漂移的核心。>

## 3. 参考实现
<可直接复制的代码片段或落地点指引。>

## 4. 反模式（禁止这样做）
<列出错误做法，作为"改动时对齐"的自查清单。>
```

### 5.1 初始条目 C01 — 删除操作的确认形式

详见 `docs/conventions/c01-delete-confirm.md`（本设计批准后随框架一起创建）。要点：

- **适用**：任何对单条或多条数据的删除操作（行内/批量/详情页）。
- **不适用**：移除表单子项、取消暂存草稿等非持久化删除。
- **决策树**：按"可逆性 + 影响范围"选——
  - 可逆（软删除/回收站）且单条 → Popconfirm
  - 不可逆但单条 → Popconfirm（文案强调不可恢复）
  - 批量（≥2 条）→ Modal（列出待删项 + 二次确认）
  - 级联影响（删父连带子）→ Modal（必须列出影响范围）
- **参考实现**：行内 Popconfirm 见 `system/role/index.tsx`。
- **反模式**：物理删除+批量却只用 Popconfirm；同类场景形式不一致；用 Modal 删单条可逆数据。

### 5.2 初始条目 C02 — 带"编号"的业务对象创建流程

详见 `docs/conventions/c02-numbered-object-create.md`。要点：

- **适用**：任何业务对象，其"编号"字段由编号引擎生成（颜色/客户/商品/计量单位等）。
- **不适用**：用户自定义的自由字段；无编号概念的对象。
- **固定流程**：
  1. 打开"新建"表单时，调用 `previewCode(targetType, categoryCode?)`（`frontend/src/api/numbering.ts`）。
  2. 预览编号写入表单"编号"字段，设为 `readOnly`。
  3. 判断返回值：
     - 非空 → 编号只读展示预览值，表单其余字段正常可用。
     - `null` → 表单 `disabled`，顶部 `Alert`（type='warning'）提示"未找到可用编号规则，请先配置编号规则"，提示中带跳转入口；确定按钮 `disabled`。
  4. 提交成功后关闭表单、刷新列表。
- **参考实现**：`frontend/src/pages/master-data/color/index.tsx`（已按此实现，是当前最规范实例）；`frontend/src/pages/business/customer/form.tsx` 为第二实例。
- **反模式**：让用户手填编号；编号字段可编辑；预览失败时表单仍可用；只灰掉表单不提示原因。

## 6. 生效机制：四道闸门

光把标准写进文件不会自动生效。痛点根因是"按需加载失败"——我只有在"意识到场景有标准"时才查，但漂移发生在"没意识到"时。机制核心目标：**把"需要我意识到"变成"不依赖我意识到"**。

```
闸门① AGENTS.md 自动注入（必然读到——解决"没看见"）
   ↓
闸门② 主决策表的"触发场景"列（必然比对——解决"没意识到该用"）
   ↓
闸门③ 详情文档的决策树（必然照做——解决"知道该用但做法漂移"）
   ↓
闸门④ "改动时对齐"规则（必然自查——解决"已有代码不符合"）
```

- **闸门①**：AGENTS.md 会话启动自动注入，零成本，机制天然支持。
- **闸门②**：主决策表的"触发场景"列必须写成**穷举式触发词**（"行内删除/批量删除"、"颜色/客户/商品创建"），而非抽象描述（"删除数据时"）。这是主决策表的维护纪律。
- **闸门③**：在 AGENTS.md 写入四条**硬约束实施规则**（见 6.1），把"读详情"变成工作流里绕不过的卡点。
- **闸门④**：详情文档的"反模式"小节 = 改动时的自查清单。把"对齐"这个抽象要求，变成"逐条比对反模式"的可操作动作。

### 6.1 写入 AGENTS.md 的实施规则（不可绕过）

```markdown
### 实施规则（不可绕过）

1. **动手前必查**：实现任何功能前，扫一遍上方主决策表的"触发场景"列。
   命中任何一条，必须先读对应详情文档，再开始写代码。
2. **改动时对齐**：修改已有功能时，检查该功能是否命中某条标准。
   若命中且现状与标准不符，顺手修正并向用户说明。
3. **新增标准**：当用户描述"期望的工作流"或指出不一致时，先把它沉淀成一条
   新的 Convention（加一行表 + 建详情文件），再实现。不要只停留在对话里。
4. **引用而非复述**：实现某功能时，在回复里引用命中的标准 ID
   （如"本次删除操作按 C01 走 Popconfirm"），方便用户核对。
```

### 6.2 可靠性边界（坦诚说明）

本机制能大幅降低漂移概率，但不能 100% 杜绝。AGENTS.md 是"注入到上下文"而非"强制执行"——最后一公里仍依赖 Agent 的执行纪律。能 100% 强制的是把检查做成 **hook**（如 pre-commit 静态扫描），但成本高且只对"能用静态规则检出"的模式有效。**建议先用"文档+纪律"跑起来，积累几条标准后，再针对高频漂移点考虑加 hook 兜底**（见 §8 渐进式 Lv3）。

## 7. 标准的生命周期

### 7.1 三种触发时机

| 触发 | 例子 | 沉淀动作 |
| --- | --- | --- |
| 用户主动提出期望 | "删除都用统一确认方式" | 立即沉淀为 Convention，再实现 |
| 用户发现不一致并指出 | "这次 Modal 上次 Popconfirm" | 先沉淀标准，再用标准修正已有代码 |
| Agent 实现中识别可复用模式 | "这个流程该统一" | **提议**给用户确认，确认后再沉淀 |

**边界**：Agent 不可未经用户确认就单方面新增标准。标准是约束，约束必须经用户同意。Agent 能做的是"识别 → 提议 → 等拍板"。

### 7.2 渐进式成长（每条标准该写多"厚"）

新增标准的门槛低到"加一行表 + 写个决策树"。细节可后补：

```
Lv0（提议，未确认）   → 只在对话里提一句，不落盘
Lv1（确认，初版）     → 主决策表加一行 + 详情只填"适用边界"和"标准（决策树）"
Lv2（验证后补全）     → 补"参考实现"（真实代码/接口名）和"反模式"
Lv3（高频漂移点）     → 考虑加 hook 兜底
```

C01、C02 因用户描述完整，初版即按 Lv2 落地。未来新条目允许先 Lv1。

### 7.3 修改已有标准（轻量版，用户确认）

采纳简化规则：标准的实质变更**直接修改详情文档**，不新建条目；在提交信息里说明变更内容。放弃"只增不改"的 ADR 风格，换取符合"边用边补"节奏的低摩擦。

> 取舍说明：ADR 风格（新建+废弃旧条目）的好处是历史可追溯，但会显著增加"边用边补"的摩擦。用户明确选择简化——直接改、提交信息说明。代价是失去决策沿革的完整留痕，权衡可接受。

## 8. 目录结构与文件命名

```
docs/conventions/
├── README.md                          ← 索引 + 如何新增一条标准的说明
├── c01-delete-confirm.md
├── c02-numbered-object-create.md
└── ...
```

- **ID**：`c` 小写 + 两位数字（c01, c02...），与主决策表 ID 一一对应。
- **文件名**：`c01-delete-confirm`（ID + 短横线名），可读且唯一。
- **目录入口**：`docs/conventions/README.md` 写清"如何新增一条标准"的步骤，让追加时不依赖记忆、照做即可。

## 9. 验收标准

框架落地后，应满足：

1. `AGENTS.md` 新增「业务与交互模式标准」章节，含主决策表（含 C01、C02 两行）+ 四条实施规则。
2. `docs/conventions/` 目录存在，含 `README.md`、`c01-delete-confirm.md`、`c02-numbered-object-create.md`。
3. 两条详情文档均含四节：适用边界 / 标准（决策树）/ 参考实现 / 反模式。
4. C02 的参考实现指向真实接口 `previewCode` 和真实文件 `color/index.tsx`、`customer/form.tsx`。
5. 客户删除的偏离点（`business/customer/index.tsx` 用 Modal.confirm）在 C01 落地后顺手对齐为 Popconfirm（或按 C01 决策树判定其正确形式），并在提交信息引用 C01。

## 10. 不在本次范围（YAGNI）

明确排除，避免范围蔓延：

- **Hook 兜底**：留待积累几条标准、识别出高频漂移点后再做（Lv3）。
- **全项目审计**：不做"扫描全项目找出所有不符合点"的工具化审计。仅靠"改动时对齐"渐进修正。
- **其他业务模式标准**（如表单提交后是否关闭弹窗、批量操作选中规则、空状态展示等）：本次只立 C01、C02 作样板，其余按"边用边补"在后续会话中追加。
- **现有代码的大规模整改**：除"客户删除偏离点"作为 C01 的即时验证外，不主动重写其他已实现功能。
