## 0. 文档性质与范围

本文档是**方向设计（directional design）**，不是实现计划。它定义工艺库模块的架构定位、核心机制、数据模型骨架、前端交互约束，以及与现有设计的差异。

本文档**修订** `docs/specs/2026-07-01-tech-stack-design.md` §4.2/§4.3 中关于"工艺作为面料开发单子单据"的定位——将工艺从子单据**提升为独立的主数据资源库**。详见 §1.2、§5.1。

本文档**不包含**：精确字段清单、API 设计、迁移脚本、前端组件实现细节（见 §5.4 待定项）。

## 1. 架构定位与分层调整

### 1.1 工艺库的新定位

工艺从"面料开发单的子单据"**升级为独立的第二层主数据资源库**。新的分层依赖关系：

```
【第一层:基础数据】
   基础设置 (编号规则 / 单位)
        ↑ 被全局引用

【第二层:主数据】
   设备管理 / 原料物料 / 客户管理 / 颜色对色 / 样品文档
   工艺库 (织造工艺 / 流程工艺 / 染色工艺)   ← ★ 新增，与其它主数据平级
        ↑ 被面料开发单引用

【第三层:核心流程】
   面料开发 (主轴)
       └── 引用工艺库中的工艺 (生成快照，打样可改，归档回写)
   产品管理 (归档后的成品)
```

**关键变化**：原来挂载在面料开发单下的三种工艺，现在前移到第二层，成为独立的主数据资源。面料开发单不再"拥有"工艺，而是**引用**工艺库。

### 1.2 对 tech-stack-design 的修订

| | 旧设计（§4.2/§4.3） | 新设计 |
|---|---|---|
| 工艺的位置 | 第三层核心流程，面料开发单的子单据 | 第二层主数据，独立资源库 |
| 工艺的来源 | 面料开发 UI 内就地新建 | 只能从工艺库引用，不可就地新建 |
| 工艺与面料开发单的关系 | 拥有（挂 FabricDevelopmentId） | 引用（快照 + fork 溯源 + 归档回写） |

> 修订清单见 §5.1。实际修订源文档在方向被批准、进入实现规划时执行，不在本次方向文档执行。

### 1.3 与 ADR-0004 的关系

ADR-0004 用"工艺归档复杂度"作为 Clean Architecture 分层的动机之一（"改织造的代码不会碰到染色的"）。新设计**不冲突**——三种工艺仍是三个独立实体（独立表、独立 Service），关注点分离的优势保留。独立化后边界更清晰：工艺库自身是一个完整的领域，不再寄生在面料开发单里。

## 2. 核心机制（8 项决策 + 3 项边界）

以下决策在方向讨论中逐条确认，构成工艺库模块的设计基线。

### 2.1 八项核心决策

| # | 决策 | 内容 |
|---|---|---|
| 1 | **快照 + 回写** | 面料开发单引用工艺库时生成**快照**；打样阶段可就地改快照（改的是副本，不碰库里的原工艺）；归档入库时，**改过的快照回写成新工艺**进库，**没改过的保持对原模板的引用**。 |
| 2 | **fork 溯源** | 回写生成的新工艺存 `SourceId` 指回它 fork 自的原工艺。库内可看出"工艺 X 派生自工艺 Y"，形成 DAG 家族图谱。提供"溯源"可视化功能（类似 Git fork 图谱）。 |
| 3 | **三个独立实体** | 织造/流程/染色三种工艺字段差异巨大，各自独立实体、独立表、独立编号。不靠继承，靠工具组件复用机制（见边界 3）。 |
| 4 | **织机参数快照** | 织造工艺卡第②部分（织机类型+织机参数）在引用时把设备运行模板的参数值**复制成快照**进工艺卡。师傅改参数只改自己这张卡的副本，模板和别的卡不受影响。整张工艺卡（含织机参数）行为统一：都是快照。 |
| 5 | **字段级 diff 驱动 `IsModified`** | 不用"是否触发过编辑"判定修改，而用"最终值是否偏离原始快照值"判定。快照生成时冻结全部原始值；保存时把当前值与原始值逐字段比对：有任何差异 → `IsModified=true`；全等 → `false`。参照系是**快照自己的原始值**，不是工艺库里原工艺的当前值（即使打样期间原工艺被改也不影响判定）。如"车速 80→80"算没改。 |
| 6 | **编号策略 B+C+E** | ① 三种工艺各自独立 targetType（`craft-weaving` / `craft-dyeing` / `craft-process`），编号互不干扰；② 只给工艺库的工艺编号，快照卡不单独编号（靠父面料开发单 + 类型定位）；③ 归档回写生成的新工艺**重新编号**（调编号引擎生成新号），fork 血缘由 `SourceId` 承载，不污染编号。 |
| 7 | **菜单 C：单入口 + 页内 Tabs** | 工艺库是侧边栏"业务资料"域下的**一个独立菜单项**（不拆三个，也不做多级 SubMenu）。点进去用**页内 Tabs** 切换三种工艺。贴合 AGENTS.md 的扁平导航规范，与编号管理（规则/字典/日志三 Tabs）同范式。 |
| 8 | **引用即快照** | 用户在面料开发单确认引用工艺的那一刻，**立即深拷贝**原工艺全部数据生成快照。不留"引用了但还没冻结"的中间态。这让 `IsModified` 的参照系天然成立，也避免打样期间原工艺被改导致快照内容飘移。 |

### 2.2 三项边界决策

| # | 边界问题 | 决策 | 理由 |
|---|---|---|---|
| 边界 1 | 快照的"深拷贝边界"画在哪 | **A：只冻结值，不冻结引用对象** | 工艺卡自己的数据（织机参数值、纱比、克重等）全冻结；对外部主数据（物料/颜色/计量单位）保持外键引用，不复制。与设备模块、物料模块的既有范式一致。 |
| 边界 2 | 工艺库原工艺被引用后能删吗 | **A：有引用即禁删 + 鼓励停用** | 工艺库的工艺是"模板资源"，删除影响面大。被任何快照（无论是否归档）引用的工艺禁止删除，引导用户改用"停用"。与"删客户有订单引用禁删"同范式，让 c01 删除规约有据可依。 |
| 边界 3 | 三种工艺实体要不要抽公共基类 | **C：实体独立，抽工具函数** | 实体保持独立（避免 EF 继承映射复杂度），但通用机制（快照深拷贝器、字段 diff 比对器、归档回写器、溯源遍历器）抽成可复用工具组件。与设备模块 `EquipmentParameterValueValidator`（静态纯函数工具类）同一思路。 |

## 3. 数据模型与快照机制

### 3.1 两个数据族：工艺库（模板）+ 工艺快照（实例）

整个设计有**两组实体**：

- **工艺库实体**（模板，第二层主数据）：织造/流程/染色各一张表，存"标准工艺"。可被多个面料开发单引用。有编号、有 `SourceId`（fork 溯源）、有状态（启用/停用）。
- **工艺快照实体**（实例，挂在面料开发单下）：面料开发单引用工艺库时生成的快照。有 `FabricDevelopmentId`（挂在哪张开发单）、`SourceCraftId`（fork 自哪条库工艺）、`IsModified`（是否真改过）、冻结的全部数据。

```
工艺库 (独立主数据)                  工艺快照 (面料开发单的子单据)
┌────────────────────┐              ┌──────────────────────────┐
│ WeavingCraft       │ ──fork──→    │ WeavingCraftSnapshot     │
│  Id, Number        │  引用即快照   │  Id                      │
│  Name              │              │  FabricDevelopmentId     │
│  SourceId (溯源)    │              │  SourceCraftId (fork自)  │
│  Status (启/停)     │              │  IsModified (字段级diff) │
│  织机参数/纱支/规格  │              │  + 冻结的全部数据副本     │
└────────────────────┘              └──────────────────────────┘
   DyeingCraft ──→ ...                 DyeingCraftSnapshot ──→ ...
   ProcessCraft ──→ ...                ProcessCraftSnapshot ──→ ...
```

> 本节聚焦织造展开。流程/染色是同构的独立实体（边界 3 = 工具复用，不靠继承）。

### 3.2 织造工艺库实体（`WeavingCraft`）

存工艺库里的"标准织造工艺"。对应织造工艺卡的 5 个部分：

| 部分 | 字段 | 说明 |
|---|---|---|
| ① 基本信息 | `Number`、`Name`、`Status`、`Remark`、`SortOrder` | 编号引擎生成（targetType=`craft-weaving`）；名称库内唯一；Status 启用/停用 |
| ② 织机参数 | 导航到 `WeavingCraftMachineParam` 子表 | 每行 = 一个参数值（设备类型+模板来源+参数名+值），**引用设备模板时复制成快照**（决策 4） |
| ③ 纱支原料表 | 导航到 `WeavingCraftYarn` 子表 | 每行 = 一种纱支（排位、名称、`ColorId` 外键、纱牌、纱批、纱比等） |
| ④ 坯布规格 | 导航到 `WeavingCraftSpec` 子表 | 每行 = 一个规格项（下机克重、下机门幅…，`UnitId` 外键） |
| ⑤ 其他信息 | 排针/列/针筒针/排纱/织胚纱长/备注 | 卡上的扁平字段或简单子表 |
| 溯源 | `SourceId` (nullable Guid) | fork 自哪条库工艺；原创为 null |

**关键点**：
- ②③④都是**值集合**（子表），随主体整表替换（同 `EquipmentTypeParameter` 的 diff 范式：PUT 时按 Id diff——null=新增、有值=更新、缺失=删除）。
- ③的 `ColorId`、④的 `UnitId` 是**外键引用**（边界 1 = 只冻结值不冻结引用对象），指向颜色库/计量单位库。
- ②的织机参数，值是**冻结的副本**（来自设备模板快照，决策 4），但"设备类型""模板来源"这两个**引用关系**保留为外键/标识，以便追溯是哪种设备哪种模板。

### 3.3 织造工艺快照实体（`WeavingCraftSnapshot`）

面料开发单引用 `WeavingCraft` 时，深拷贝生成的实例：

| 字段 | 说明 |
|---|---|
| `Id`、`FabricDevelopmentId` | 挂在哪张面料开发单下 |
| `SourceCraftId` | fork 自工艺库的哪条（`WeavingCraft.Id`） |
| `IsModified` | 字段级 diff 驱动（决策 5）：当前数据 vs 快照生成时冻结的原始数据 |
| `OriginalValuesJson` | **冻结的原始数据快照**，作为 diff 的参照系（决策 5 的物理载体） |
| + 织机参数/纱支/规格/其他信息的**完整副本** | 与库实体的子表结构一致，但是独立的物理行 |

**机制落地**：
- **引用即快照**（决策 8）：创建快照时，一次性深拷贝 `WeavingCraft` 的全部子表数据进 `WeavingCraftSnapshot`，同时把这份原始数据序列化存入 `OriginalValuesJson`。
- **字段级 diff**（决策 5）：每次保存快照时，把当前数据与 `OriginalValuesJson` 反序列化比对。任一字段不同 → `IsModified=true`；全等 → `false`。
- **归档回写**（决策 1）：面料开发单归档时，`IsModified=true` 的快照 → 调编号引擎生成新 `WeavingCraft`，`SourceId` 指向原工艺；`IsModified=false` 的快照 → 不回写，开发单的 `SourceCraftId` 直接指向原工艺。

### 3.4 fork 溯源的物理结构

每个库实体的 `SourceId` 形成 DAG（有向无环图）：

```
ZZ-0001 (原创, SourceId=null)
  ├── fork → ZZ-0002 (SourceId=ZZ-0001)  打样改过，回写
  │            ├── fork → ZZ-0005 (SourceId=ZZ-0002)
  │            └── fork → ZZ-0006
  └── fork → ZZ-0003 (SourceId=ZZ-0001)  打样改过，回写
                └── fork → ZZ-0007
```

"溯源图谱"功能 = 沿 `SourceId` 反向 BFS 遍历，渲染为树/DAG。`SourceId=null` 是根（原创工艺）。

### 3.5 三种工艺的关系：独立实体 + 共享工具

三种工艺各有一对实体（库 + 快照），共 6 张主表 + 各自子表。它们共享的不是表结构，而是**机制**——这些机制抽成可复用工具组件（边界 3 = 工具复用）：

| 共享机制 | 落地为工具组件 | 类比项目既有 |
|---|---|---|
| 深拷贝生成快照 | `CraftSnapshotGenerator<TCraft>` | — |
| 字段级 diff 驱动 `IsModified` | `CraftSnapshotDiffChecker<TSnapshot>` | `EquipmentParameterValueValidator`（静态纯函数） |
| 归档回写 + 重新编号 | `CraftArchiveWriter<TCraft, TSnapshot>` | — |
| fork 溯源遍历 | `CraftLineageQuery<TCraft>` | — |

三种工艺的 Service 调用这些工具，各自维护自己的子表结构。

## 4. 生命周期与状态流转

### 4.1 工艺库里一条工艺的生命周期

```
                  【诞生：两种来源】
                         │
        ┌────────────────┴────────────────┐
        │                                 │
   ① 原创建立                         ② 归档回写诞生
   (工艺库 UI 直接新建)                (面料开发单打样改过→归档)
   SourceId = null                    SourceId = 指向被 fork 的原工艺
   状态: 启用                          状态: 启用
        │                                 │
        └────────────────┬────────────────┘
                         │
                         ▼
                 【存在：可被引用】
                 被面料开发单引用 (生成快照)
                 可被多次引用 (一对多)
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
        启用 (Active)            停用 (Inactive)
        可被新开发单引用          不出现在引用选择器里
        旧引用仍有效              已有快照不受影响
```

**状态只有两个：启用 / 停用**。没有"草稿"——工艺库里的工艺一旦创建就是可用的标准工艺（回写诞生的工艺同理）。停用只是"不再被新引用"，已存在的引用和快照都不受影响（与边界 2 = 有引用禁删配套：要"去掉"一条工艺，停用它，而不是删）。

### 4.2 一张快照的生命周期（核心流程）

把决策 1/2/4/5/6/8 串成完整时间线：

```
面料开发单 FB-001 (状态: 开发中)
   │
   │ ① 用户点"引用织造工艺"，选了 ZZ-0001
   │   → [引用即快照, 决策8]
   │   → 深拷贝 ZZ-0001 全部数据 → 生成 WeavingCraftSnapshot
   │   → SourceCraftId = ZZ-0001
   │   → IsModified = false
   │   → OriginalValuesJson = 冻结的原始数据
   ▼
快照卡 (开发中, 可编辑)
   │
   │ ② 用户在打样阶段编辑这张卡
   │   (改纱比/调克重/调织机参数...)
   │   → 保存时: [字段级 diff, 决策5]
   │     比对 [当前数据] vs [OriginalValuesJson]
   │     有差异 → IsModified = true
   │     无差异(如80→80) → IsModified = false
   │   → 改的是快照副本, 工艺库的 ZZ-0001 不受影响 [决策1]
   ▼
快照卡 (编辑后, IsModified=true 或 false)
   │
   │ ③ 面料开发单走状态流转: 开发中 → 待评审 → 归档
   │   归档触发回写判定:
   │
   │   ┌─ IsModified = true (真改过) ──────────────┐
   │   │  → [归档回写, 决策1+6E]                    │
   │   │  → 调编号引擎生成新工艺 ZZ-0099            │
   │   │  → ZZ-0099.SourceId = ZZ-0001 [fork,决策2] │
   │   │  → 快照的 SourceCraftId 改指向 ZZ-0099     │
   │   │  → 工艺库 +1 条新工艺                      │
   │   └────────────────────────────────────────────┘
   │   ┌─ IsModified = false (没真改) ─────────────┐
   │   │  → 不回写                                 │
   │   │  → 快照的 SourceCraftId 保持 = ZZ-0001    │
   │   │  → 开发单归档后指向原工艺                  │
   │   └────────────────────────────────────────────┘
   ▼
面料开发单 FB-001 (状态: 已归档)
   → 面料信息 + 工艺快照 全部锁定只读
   → 生成产品记录进产品管理 (tech-stack §4.3 归档语义)
```

### 4.3 一个开发单引用多种工艺

按现有设计（§4.3），一张面料开发单可同时引用织造/流程/染色三种工艺。每种独立走 §4.2 的生命周期：

```
FB-001 (开发中)
 ├── WeavingCraftSnapshot  (SourceCraftId=ZZ-0001, IsModified=true)  → 归档回写成 ZZ-0099
 ├── ProcessCraftSnapshot  (SourceCraftId=LC-0001, IsModified=false) → 保持指向 LC-0001
 └── DyeingCraftSnapshot   (SourceCraftId=RS-0001, IsModified=true)  → 归档回写成 RS-0050
```

三种工艺**互不依赖、独立判定、独立回写**（继承 §4.3 的"并列挂载"语义，只是挂载方式从"拥有"变成"快照引用"）。

### 4.4 归档的不可逆性

面料开发单一旦归档，其下的所有快照卡**锁定只读**（tech-stack §4.3 已定）。这意味着：
- 归档后 `IsModified` 不再变化（不再编辑）。
- 归档后 `SourceCraftId` 固定（回写已完成或确定为原工艺）。
- 即使后来工艺库的原工艺被停用，已归档快照展示的内容不变（因为它存的是自己的数据副本，只是 `SourceCraftId` 指向一个可能已停用的工艺——展示时需要能解析到已停用的工艺信息）。

## 5. 前端导航与交互

### 5.1 工艺库的菜单定位（决策 7 落地）

按"单入口 + 页内 Tabs"（C 方案），工艺库是侧边栏**业务资料**域下的一个独立项：

```
侧边栏
├── 生产流程
│     └── 面料开发 (未实现)
├── 业务资料
│     ├── 客户管理
│     ├── 物料管理
│     ├── 工序管理
│     ├── 设备管理
│     └── 工艺库  ← ★ 新增独立项
├── 基础设置
│     ├── 编号管理
│     └── 计量单位
└── 系统管理
```

点进"工艺库"，是一个**列表页 + 顶部 Tabs**：

```
┌─────────────────────────────────────────────────────────┐
│  [织造工艺] [流程工艺] [染色工艺]      ← 页内 Tabs       │
```

- **溯源**操作：行内按钮，点开 fork 家族图谱（DAG 可视化）。

### 5.2 工艺库的 CRUD 交互

| 操作 | 交互 | 约束 |
|---|---|---|
| 新建 | 弹出表单/抽屉，填 5 个部分的工艺卡 | 编号引擎预览（`useNumberingPreview('craft-weaving')` + `<CategorySelect>`，走 c02 标准）；保存时调编号引擎 |
| 编辑 | 同新建表单回填 | 启用/停用都可编辑 |
| 查看 | 只读详情页 | 展示 5 部分 + 溯源链 |
| 停用/启用 | 行内开关或操作项 | 即时生效 |
| 删除 | Popconfirm（按 c01） | **有引用禁删**（边界 2 = A），引导停用 |
| 溯源 | 打开图谱抽屉/页 | 沿 `SourceId` 渲染家族 DAG |

> 新建/编辑表单是工艺库最复杂的前端——5 个部分中，②织机参数（要引用设备模板并展示参数）、③纱支表（动态行 + 颜色选择）、④坯布规格（动态行 + 单位）都是动态表格组件。这块在前端实现时工作量最大，标记为"前端重点"。

### 5.3 面料开发单引用工艺的交互（预演）

虽然面料开发模块还没做，但本方向文档预先定义"引用"这个交互长什么样，因为它决定工艺库怎么被消费。等面料开发模块设计时，这部分作为约束输入：

```
面料开发单 FB-001 详情页 (开发中状态)
├── 基本信息 tab
├── 织造工艺 tab  ← 点开
│     ├── [当前: 无引用]
│     └── [从工艺库引用] 按钮  ← 唯一入口
│           → 弹出工艺库选择器 (只列 启用 的织造工艺)
│           → 选定 ZZ-0001, 确认
│           → 后端: 引用即快照, 生成 WeavingCraftSnapshot
│           → tab 内容变成: 快照卡 (可就地编辑)
│     └── [更换工艺] [查看溯源] (快照生成后)
└── 流程工艺 tab / 染色工艺 tab (同结构)
```

**关键交互约束（决策 1 落地）**：
- 面料开发页**没有"新建工艺"按钮**——只能"从工艺库引用"。这是"工艺库独立化"最直接的前端体现。
- 引用后生成的快照卡**可就地编辑**（打样迭代），但改的是快照，工艺库不动。
- 快照卡上有视觉标识表明"这是引用自 ZZ-0001 的快照，已修改/未修改"。

### 5.4 溯源图谱（决策 2 落地）

"Git fork 图谱"作为工艺库的一个可视化功能：
- 入口：工艺库列表行的"溯源"按钮、工艺详情页的"溯源"tab。
- 内容：以当前工艺为焦点，沿 `SourceId` 向上找祖先、向下找后代，渲染成树/DAG。
- 选中的工艺节点可点击跳转到该工艺详情。
- 这个组件是**独立的可视化组件**，三种工艺共用（参数化实体类型）。

> 实现层面，DAG 图谱可能需要前端图表库（如 G6/AntV）。这是前端的一个技术选型点，标记为"实现时调研"，不在本方向决策范围内。

### 6.2 代码层面：新增与不变

**新增（未来实现时）**：
- Domain：6 个实体（3 库 + 3 快照）+ 各自子表实体 + `CraftStatus` 枚举
- Application：3 套 Service（库 CRUD + 快照生命周期）+ 4 个共享工具组件（`CraftSnapshotGenerator` / `CraftSnapshotDiffChecker` / `CraftArchiveWriter` / `CraftLineageQuery`）
- Infrastructure：EF 配置 + Migration
- Api：控制器 + DI 注册 + 权限策略（`craft-weaving:read/create/update/delete` 等）+ 编号引擎新 targetType 注册
- Frontend：工艺库列表页（Tabs）+ 工艺卡新建/编辑表单 + 溯源图谱组件 + 面料开发页的"引用工艺"交互（后者依赖面料开发模块）

**完全不变**：
- 设备模块（`EquipmentTemplate` 等已实现）—— 工艺卡只是**读取并复制**设备模板的参数，不改设备模块任何代码
- 编号引擎、权限框架、删除规约（c01）、编号对象创建标准（c02）—— 工艺库直接复用，作为新的消费者
- 现有的主数据（物料/颜色/计量单位）—— 工艺卡以**外键引用**它们，不冻结

### 6.3 前置依赖（决定实现顺序）

按依赖关系，实现工艺库之前必须就绪的：

1. **设备管理收尾**（当前分支 `feat/equipment-mgmt`）—— 织机参数快照依赖 `EquipmentTemplate` 的参数结构稳定。这是为什么"先收尾设备"是合理的前置。当前设备管理后端约 70%、前端 0%，且 DI/权限/控制器全缺，需先收尾。
2. **编号引擎接入点**（已就绪）—— 加 `craft-weaving` / `craft-dyeing` / `craft-process` 三个 targetType 即可。
3. **可选但强烈建议：面料开发主轴**—— 工艺库**本身**（CRUD + 溯源）可以独立先做，不依赖面料开发单。但**快照生命周期**（引用→打样→回写）必须有面料开发单作为宿主才能端到端跑通。

可行的实现策略有两种（不在本方向文档拍板，留待实现规划）：

- **策略甲**：先做工艺库的独立部分（库 CRUD + 溯源），再做面料开发主轴，最后接通快照生命周期。**工艺库能尽早产出价值**。
- **策略乙**：先做面料开发主轴，再做工艺库，一次性接通。**端到端流程连贯**。

### 6.4 本次方向文档明确"不做决定"的待定项

这些是实现阶段才需要细化的，本方向文档只记录、不决策：

- 工艺卡 5 个部分里每个子表的**精确字段清单**（纱支表到底有哪些列、坯布规格表有哪些项）—— 需要业务确认
- 流程工艺、染色工艺的**具体字段结构**（本次只确认了织造）—— 需要业务输入
- 溯源图谱的**前端图表库选型**（G6 / AntV / 自绘）
- `OriginalValuesJson` 的**序列化格式**与 diff 算法的精确实现
- 归档回写时"新工艺命名"规则（自动 `ZZ-0001 的变体` 还是要求用户命名）
- 工艺库的**权限粒度**（是否区分"工艺工程师可编辑/其他人只读"等角色细分）
