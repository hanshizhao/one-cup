# 设备类型表单页布局重构设计

> 日期：2026-07-07
> 状态：已批准（待实现）
> 范围：设备类型（EquipmentType）新建/编辑表单页的布局重构

---

## 1. 背景与目标

设备类型表单页已完成的改造（返回 bug 修复 + 官方范式结构）解决了**结构问题**，但布局仍有两个**视觉体验问题**：

1. **宽屏下控件过宽**：无论基础信息用单列纵排还是 3 列栅格，在 1920px 宽屏下控件要么被拉得过长（单列），要么留白严重（3 列）。控件宽度应有合理约束。
2. **参数编辑器预览面板价值低**：「在模板里长什么样」预览面板口语化、玩具感，占 300px 宽度挤压参数卡片，且对「定义参数」核心任务无帮助。

**目标**：改为**左右分栏布局**（基础信息左 + 参数定义右），既约束控件宽度，又充分利用宽屏横向空间；同时删除预览面板。

---

## 2. 决策汇总

| 决策点 | 结论 |
|---|---|
| 整体布局 | 左右分栏，基础信息在左、参数定义在右 |
| 宽度比例 | 左 35% : 右 65%，gap 16px |
| 左栏（基础信息）内部 | 顶标签单列（`layout="vertical"`），6 字段纵向流 |
| 右栏（参数定义） | 删除预览面板，参数卡片占满栏宽 |
| 两侧对齐 | `align-items: flex-start`（顶部对齐，各自独立延伸） |
| 响应式 | 窄屏（<1200px）左右堆叠回上下 |
| 底部操作栏 | 保持现有 sticky 底部栏（不变） |

### 2.1 为什么是左右分栏

- **约束宽度**：分栏后每栏只占部分宽度，控件不再被拉到超宽
- **内容匹配**：左栏 6 个短字段（顶标签单列）高度 ≈ 右栏 3-4 张参数卡片，两侧高度相对接近
- **空间利用**：宽屏横向空间被充分利用，而非单列傻长
- **主次分明**：参数定义是设备类型的核心内容（动态、内容多），给宽栏；基础信息是元数据（固定、简短），给窄栏

### 2.2 为什么删除预览面板

- 「定义参数」时用户关心的是 min/max/precision 等约束值本身，不是「下游渲染成什么样」
- 预览面板对核心任务无帮助，却占 300px 宽度和视觉注意力
- 口语化文案「在模板里长什么样」不够专业
- 删除后参数卡片占满栏宽，min/max/精度/单位四控件更宽敞

## 3. 布局规范

### 3.1 整体结构

```
面包屑 + 页头（标题+副标题）
┌──────────────────────────────────────────────┐
│ ┌─ 基础信息 ────┐  ┌─ 参数定义 ───────────┐  │
│ │ (35%, 顶标签) │  │ (65%, 卡片占满)       │  │
│ │ 6 字段单列    │  │ 统计条 + 卡片列表     │  │
│ └───────────────┘  └───────────────────────┘  │
└──────────────────────────────────────────────┘
[底部 sticky 操作栏：取消/保存]
```

### 3.2 左栏 · 基础信息

- `Form layout="vertical"`（顶标签）
- 6 个字段，每个独占一行（无内部网格）：
  1. 类型编号（只读 Input，新建时显示编号预览）
  2. 类型名称 *（Input，必填）
  3. 编号分类码（CategorySelect，`!editing && preview.includeCategory` 条件渲染）
  4. 排序号（InputNumber）
  5. 启用状态（Switch）
  6. 备注（TextArea，全宽）
- Card 内 `Typography.Title heading={6}` 标题「基础信息」

### 3.3 右栏 · 参数定义

- Card 内 `Typography.Title heading={6}` 标题「参数定义」+ 统计后缀（如「8 项 · 拖拽排序 · 实时校验」）
- **统计条**保留（总数/数值/枚举/文本/待修正）
- **参数卡片列表**：占满栏宽，保留全部现有功能（拖拽排序、类型切换、必填开关、实时校验、删除）
- **预览面板删除**：移除 `preview-panel`、`selectedKey` state、`selectedParam`、卡片选中态（`onSelect`/`is-selected`）
- 「+ 添加参数」按钮保留

### 3.4 响应式（窄屏降级）

- 宽屏（≥1200px）：左右分栏 35:65
- 窄屏（<1200px）：左右堆叠为上下（基础信息在上，参数定义在下）
- 用 CSS `@media (max-width: 1199px)` 实现，flex 方向从 `row` 改 `column`

### 3.5 底部操作栏（不变）

- 保持现有 `form-page-actions-bar`（sticky, bottom:0, 顶部边框+阴影, 右对齐 取消/保存）

## 4. 改动清单

| 文件 | 改动 |
|---|---|
| `type/TypeFormPage.tsx` | 基础信息 + 参数定义两个 Card 包进左右分栏容器（新 class `form-split-layout`）；基础信息内部从 3 列网格改为顶标签单列（`layout="vertical"` 无 Row/Col） |
| `type/ParameterEditor.tsx` | 删除预览面板（`preview-panel` 整块 JSX）、`selectedKey` state、`selectedParam`、`onSelect`/选中态逻辑；`editor-layout` 从左右分栏改为单列 |
| `style/index.module.less` | 新增 `.form-split-layout`（flex row, gap 16, 窄屏降级 column）；`editor-layout`/`preview-*` 相关 class 清理（预览删除后无用） |

### 4.1 不需要改动的文件

- `index.tsx`（Tab query 持久化）、`router.tsx` —— 不动
- `api/equipment.ts`、`useNumberingPreview`、`CategorySelect` —— c02 资产原样复用
- 底部操作栏样式 —— 保持
- 参数卡片的拖拽/类型切换/校验/统计条逻辑 —— 全部保留

### 4.2 locale key 处理

- 预览相关 key（`equipment.type.param.preview.*`）：代码不再引用，但**保留在 locale 文件**（删除 key 需确认无其它引用，保守起见保留，无害）

## 5. c02 编号流程（不变）

- 类型编号：编辑显示实际 code，新建显示 `preview.code`（只读 Input）
- 分类码：`!editing && preview.includeCategory` 时条件渲染
- 提交：`createEquipmentType({ ...payload, categoryCode: preview.categoryCode })`
- 无规则：`preview.noRule` 时禁用表单 + 顶部警告 Alert + 保存按钮 disabled

## 6. 验收要点

1. 宽屏下表单呈**左右分栏**（基础信息左 35%、参数定义右 65%），控件宽度合理，不再过宽或留白
2. 左栏基础信息：顶标签单列，6 字段，控件等宽占满栏宽
3. 右栏参数定义：**无预览面板**，统计条 + 卡片列表占满栏宽
4. 参数卡片功能正常：拖拽排序、类型切换（数值/枚举/文本）、必填开关、min/max/精度/单位、实时校验、删除
5. 两侧顶部对齐；参数多时右栏自然向下延伸，左栏固定
6. 窄屏（窗口宽<1200px）下左右**堆叠为上下**
7. 底部 sticky 操作栏正常（取消/保存，无规则时保存禁用）
8. 返回/取消正确跳回设备类型 Tab（上一轮已修复，不回归）
9. c02 编号预览/分类码/无规则禁用全流程正常
10. `npm run build` 通过
